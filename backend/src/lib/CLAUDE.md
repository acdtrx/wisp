# Backend Ops Rules (storage, cloudInit, paths)

- **paths.js:** Wisp app-glue path helpers and the **API-input security gate** for VM file paths. Exports: `getVMBasePath(name)`, `getImagePath()`, `ensureImageDir()`, `resolveLibraryPath(p)` (relative→absolute via image library), and `assertPathInsideAllowedRoots(absPath, vmName)` (rejects paths outside `imagePath` and per-VM dir). **Routes resolve+validate user-supplied paths here, before passing absolute paths to vmManager.** vmManager itself is policy-agnostic about which host paths a deployment considers "allowed."
- **Strict managers:** vmManager and containerManager carry **zero Wisp-glue imports** — no `routeErrors`, `validation`, `cloudInit`, `vmMdnsPublisher`, `sections`, `paths`, `config`, `settings`, `atomicJson`, `bootCleanup`, jobs, or downloads. Each owns its private error factory (`vmError` / `containerError`, inlined in `vmManagerShared.js` / `containerManagerConnection.js`), its private validators (`vmManagerValidation.js` / `containerValidation.js`), and its private path helpers / atomic-write copy. Generic ISO ops (`attachISO` / `ejectISO`) carry `{ createIfMissing }` / `{ removeSlot }` opts so cloud-init's special slot handling rides the public primitive — the orchestration lives in `lib/cloudInit.js` (Wisp glue), not in vmManager. Future changes to vmManager / containerManager are appropriate only for **generic libvirt/containerd functionality**; Wisp-specific orchestration (cloud-init flow, mDNS publish hooks, section assignments, file-cleanup rollbacks) goes in routes or app-glue files. Full rule + rationale in `docs/WISP-RULES.md` § Architecture.
- **Manager configure() (push model):** Boot wiring in `backend/src/index.js` calls `vmManager.configure({ vmsPath })` and `containerManager.configure({ containersPath, resolveMount })` before any route registers. The `resolveMount` callback is a sync `(id) => null | { id, mountPath, label? }` over `getConfigSync().mounts`. Internal helpers: `lib/linux/vmManager/vmManagerPaths.js` (vmManager-private, holds `getVMBasePath`/`getVmsPath` only — image-library policy stays in Wisp glue); `lib/linux/containerManager/containerPaths.js` (containerManager-private, holds path getters + `resolveMount` slot); `lib/linux/containerManager/atomicJson.js` (vendored copy of `lib/atomicJson.js`, kept private to break the policy edge ahead of eventual lib extraction). The `oci-image-meta.json` sidecar lives at `<containersPath>/.oci-image-meta.json` (containerManager-owned cache).
- **Disk copy + resize (storage/diskOps.js, exposed via `storage/index.js`):** `qemu-img convert -O qcow2 <src> <dst>` then `qemu-img resize <dst> <size>G` if needed. Destination is the per-VM directory. Progress via child process stdout with `-p` flag; stream to client via SSE. Source file never modified or moved.
- **Cloud-init (cloudInit.js):** Wisp-glue orchestrator — owns ISO build (cloud-localds / genisoimage), config persistence, and attach/detach via `vmManager.attachISO(name, 'sde', isoPath, { createIfMissing: true })` and `vmManager.ejectISO(name, 'sde', { removeSlot: true })`. Password hashed via `openssl passwd -6` (child process). Stored under per-VM directory (`getVMBasePath`). GitHub key fetch is server-side only; frontend never calls github.com. **Routes drive the create-time flow:** `routes/vms.js` POST builds the cloud-init ISO via `generateCloudInit(name, spec.cloudInit)` *before* calling `vmManager.createVM`, and on failure cleans up via `deleteCloudInitISO(name)`. vmManager itself never imports cloudInit.js.

# vmManager Rules

These rules apply to `vmManager.js`, `vmManagerShared.js`, and files under `linux/vmManager/` and `darwin/vmManager/`.

- **Only DBus/libvirt caller:** No route or other module imports `dbus-next` for libvirt or calls libvirt except through `vmManager`. Linux libvirt logic lives under `linux/vmManager/`; the `vmManager.js` facade re-exports the platform implementation. (Avahi uses `dbus-next` in `lib/mdns/linux/avahi.js` only.)
- **Domain object paths:** `/org/libvirt/domains/<uuid>`. UUIDs from ListDomains. Use `getDomainObjAndIface(path)` — it caches the proxy in `connectionState.domainProxyCache` and skips the per-call DBus Introspect. Cache is cleared on bus disconnect and on `VIR_DOMAIN_EVENT_UNDEFINED`. Do not stash proxy references elsewhere; do not cache across reconnects.
- **VNC port:** Call `org.libvirt.Domain.GetXMLDesc(0)`, parse `<graphics type='vnc' port='...'>` from the XML. Expose via a function used by the console route; the route bridges TCP (`node:net` createConnection) to WebSocket. No third-party TCP proxy.
- **UEFI firmware:** Scan known firmware paths at startup; expose via GET /api/host/firmware. Paths vary by distro (e.g. Ubuntu /usr/share/OVMF, Fedora /usr/share/edk2/ovmf).
- **Dual CDROM:** Every VM XML defines sdc and sdd at creation, even if empty. Empty CDROM: no `<source>`. Attach/eject via `org.libvirt.Domain.UpdateDevice(xmlString, flags)` with `flags = 3` (LIVE + CONFIG).
- **CPU topology:** Always full topology in domain XML. Never `<vcpu>N</vcpu>` alone. Emit `<vcpu placement='static'>N</vcpu>` and `<cpu mode='host-passthrough' check='none' migratable='on'><topology sockets='1' dies='1' cores='N' threads='1'/></cpu>`. Same at VM creation and in updateVMConfig.
- **Live vs offline:** updateVMConfig uses UpdateDevice(xml, flags) or DomainDefineXML. Use `flags = 3` for hot-applicable changes, `flags = 2` for config-only. Return `{ requiresRestart: true }` for firmware, machine type, CPU count changes.
